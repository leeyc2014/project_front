1) 발표 목표와 핵심 메시지
==================================================
- 이 서비스는 EPCIS 물류 데이터를 기반으로 이상 징후를 조기에 탐지하고, 조치 보고와 운영 관리를 한 화면 흐름으로 연결한 웹 서비스입니다.
- 프론트엔드 관점의 핵심은 다음 3가지입니다.
  1. 실시간에 가까운 데이터 시각화(지도 + 차트 + 리스트)
  2. 운영자가 바로 행동할 수 있는 UI(조치 요청, 회원/바코드 관리, 업로드)
  3. URL/상태/권한 기반으로 끊김 없는 업무 흐름

3) 발표 대본 (5분 압축본)
==================================================
[오프닝]
안녕하세요. 프론트엔드를 담당한 OOO입니다.
저희 LOGIFLOW는 물류 이벤트를 업로드하고, 이상 징후를 시각화하며, 조치 리포트까지 연결하는 공급망 분석 웹 서비스입니다.
지금부터 로그인부터 관리 기능까지 실제 운영 흐름으로 시연하겠습니다.

[1. 로그인]
먼저 로그인 화면입니다.
아이디/비밀번호를 입력하면 백엔드 로그인 API와 통신하고, 발급된 토큰을 쿠키/세션에 저장한 뒤 대시보드로 이동합니다.
권한 정보는 프론트 전역 상태에 저장해 이후 메뉴 접근 제어에 사용합니다.

[2. 대시보드]
대시보드는 지도, 시리얼 리스트, KPI 차트 3축으로 구성했습니다.
지도는 물류 이동 경로를 Arc 형태로 보여주고, 색상으로 안전/주의/위험 수준을 직관적으로 표현합니다.
오른쪽 시리얼 리스트에서 특정 EPC를 선택하면 이벤트 타임라인을 확인할 수 있고,
좌측 필터 패널과 상단 날짜 퀵 필터로 조회 조건을 즉시 바꿀 수 있습니다.
또한 URL 쿼리와 필터 상태를 동기화해서 새로고침/공유 시에도 같은 조건을 재현할 수 있습니다.

[3. 업로드]
업로드는 관리자 기준으로 상단 위젯에서 수행합니다.
CSV 파일을 올리면 백엔드 업로드 스트림을 읽어 진행률과 로그를 실시간 표시하고,
완료 시 요약 결과(위험/주의/무결성 오류 등)를 모달로 보여줍니다.
업로드가 끝나면 대시보드 리로드 트리거를 발생시켜 차트/지도 데이터가 즉시 갱신됩니다.

[4. 진단 리포트]
진단 리포트 화면에서는 기간(주간/월간)과 제품을 선택해 분석 단위를 정하고,
생성 가능한 기간 리스트를 확인할 수 있습니다.
미리보기 버튼을 누르면 별도 출력 페이지에서 요약 통계, 유형별 KPI, 허브/이벤트 타입 분석,
상세 이상 목록까지 한 번에 확인하고 인쇄할 수 있습니다.

[5. 이상 징후 보고서]
이상 징후 보고서 메뉴에서는 조치 요청으로 등록된 리포트를 목록으로 조회합니다.
상세 모달에서 이벤트 정보, EPC 정보, 이상 내용을 확인하고,
상세 내용/처리 결과/완료 여부를 수정하여 저장할 수 있습니다.
즉, 탐지 이후의 후속 조치 이력을 같은 서비스 안에서 관리할 수 있습니다.

[6. 회원 관리]
회원 관리는 관리자 전용 기능입니다.
회원 목록 조회, 신규 등록, 권한/활성 상태 수정이 가능하고,
현재 로그인한 본인 계정 수정 시에는 안전하게 변경 범위를 제한했습니다.
운영 보안 관점에서 권한 분리와 활성화 제어를 프론트 UX에 녹여냈습니다.

[7. 바코드 관리]
바코드 관리에서는 EPC 목록을 조회하고 신규 EPC를 등록할 수 있습니다.
회사-제품-로트 선택을 연동해 잘못된 조합 입력을 줄였고,
상세 모달에서 QR 코드를 생성해 PNG로 다운로드할 수 있어 현장 라벨링/재발급에도 바로 활용 가능합니다.

[기술 스택 + 마무리]
프론트는 Next.js App Router, React 19, TypeScript 기반입니다.
UI는 Tailwind CSS, 상태관리는 Jotai를 사용했고,
시각화는 MapLibre + deck.gl(지도), ApexCharts(통계 차트), qrcode.react(QR 생성)로 구성했습니다.
핵심은 단순 조회 화면이 아니라, 데이터 업로드부터 탐지, 조치, 관리까지 운영 사이클을 프론트에서 끊김 없이 연결했다는 점입니다.
이상으로 발표 마치겠습니다.

4) 기능별 시연 체크리스트 (실제 클릭 순서)
==================================================
[로그인]
- 화면: / (로그인 페이지)
- 동작: ID/PW 입력 → 로그인 버튼
- 확인 포인트: 대시보드 이동 성공, 상단 사용자명/메뉴 표시

[대시보드]
- 화면: /dashboard
- 동작:
  1) 날짜 퀵 필터 변경
  2) 상태 필터(ALL/SAFE/CAUTION/DANGER) 클릭
  3) 시리얼 하나 열어 이벤트 타임라인 확인
  4) 지도 노드/경로 클릭으로 위치 기반 필터 적용
- 확인 포인트: 지도/차트/리스트 동시 반응

[업로드]
- 화면: 대시보드 상단 Header Upload Widget(관리자)
- 동작: 파일 선택 → 업로드/분석 시작
- 확인 포인트: 진행률 %, 로그 스트림, 완료 요약 모달, 대시보드 자동 갱신

[진단 리포트]
- 화면: /dashboard/report
- 동작: 기간(from~to), 제품, 주간/월간 선택 → 특정 행 미리보기
- 확인 포인트: /print 페이지에서 분석 보고서 출력형 화면 확인

[이상 징후 보고서]
- 화면: /dashboard/anomaly
- 동작: 목록에서 보기 → 상세 모달 → 수정 모드 → 완료 여부 토글 + 저장
- 확인 포인트: 저장 후 목록에 상태 반영

[회원 관리]
- 화면: /dashboard/members (관리자)
- 동작: 등록 버튼으로 신규 계정 생성, 기존 계정 수정
- 확인 포인트: 권한(ADMIN/USER), 활성/비활성 상태 변경 반영

[바코드 관리]
- 화면: /dashboard/manage
- 동작: 등록 버튼 → Company/Product/Lot/Serial/Date 입력 후 저장 → 상세 보기
- 확인 포인트: QR 코드 렌더링 및 PNG 다운로드

5) 프론트엔드 기술 스택 발표 파트 (1분 내외)
==================================================
[아키텍처]
- Next.js 16 App Router 기반으로 페이지 라우팅을 구성했습니다.
- 인증 토큰은 쿠키/세션에서 읽고, 공통 유틸로 Authorization 헤더에 주입합니다.

[상태 관리]
- Jotai를 사용해
  1) 로그인 사용자 정보(loginUserAtom)
  2) 업로드 후 대시보드 강제 리프레시 트리거(dashboardReloadTriggerAtom)
  를 관리했습니다.
- URL 쿼리와 필터 상태를 동기화해 재현 가능한 분석 화면을 만들었습니다.

[데이터 통신]
- fetch 기반으로 API 호출을 구성했고,
- 대시보드/차트는 토큰 지연 발급이나 일시 오류를 고려해 재시도 로직을 넣었습니다.
- 업로드는 ReadableStream을 읽어 실시간 진행 로그를 표시합니다.

[시각화]
- 지도: maplibre-gl + deck.gl ArcLayer/ScatterplotLayer
- 경로 패턴 애니메이션: 커스텀 PatternArcLayer(shader inject)
- 차트: react-apexcharts
- QR: qrcode.react

[UI/개발 생산성]
- Tailwind CSS v4로 빠르게 일관된 디자인 시스템 적용
- TypeScript 타입 정의로 API 스키마 변동에 대한 방어 로직 강화

6) 기술 포인트(심화 멘트, 시간 여유 시)
==================================================
- 대시보드 데이터는 단순 테이블이 아니라 지도/차트/리스트의 멀티 뷰로 동시 분석 가능
- 지도 자동 줌은 경로 범위와 위험도/흐름량을 반영한 적응형 로직으로 구현
- 이상 징후는 "탐지"에서 끝나지 않고 "조치 요청 등록 → 처리 결과/완료 관리"까지 추적
- 관리자 메뉴는 role 기반으로 노출 제어하여 운영 권한 분리
- EPC 등록 시 회사-제품-로트 연관 관계를 프론트에서 먼저 제한해 데이터 품질 강화

7) 예상 질문 & 답변
==================================================
Q1. 왜 상태관리를 Redux가 아니라 Jotai로 했나요?
A1. 전역 상태 범위가 크지 않고, 로그인 사용자 정보와 업로드 갱신 트리거처럼 원자 단위 상태가 중심이라 Jotai가 더 단순하고 유지보수가 쉬웠습니다.

Q2. 대시보드 필터를 URL에 동기화한 이유는?
A2. 같은 분석 조건을 링크로 공유하고, 새로고침 후에도 동일 상태를 복원하기 위해서입니다. 운영 분석 화면에서 재현성이 중요했습니다.

Q3. 업로드 진행률은 어떻게 처리하나요?
A3. 업로드 응답 스트림을 line 단위로 읽어 percent/log를 즉시 반영합니다. 완료 시 요약 통계를 모달로 제공합니다.

Q4. 권한 제어는 어디서 하나요?
A4. 로그인 시 받은 사용자 role을 기반으로 관리자 메뉴를 숨기고, 각 페이지에서도 role 체크 후 접근을 제한합니다.

Q5. 지도 시각화 선택 이유는?
A5. 물류는 위치/이동이 핵심이라 표만으로는 이상 흐름을 파악하기 어렵습니다. Arc 기반 경로 시각화가 위험 구간을 가장 빠르게 인지하게 해줍니다.

8) 시연 리허설 체크(발표 직전)
==================================================
- .env의 NEXT_PUBLIC_BACKEND_BASE_URL 확인
- 테스트 계정 2개 준비(ADMIN, USER)
- 업로드용 샘플 CSV 준비(정상+이상 이벤트 포함)
- 진단 리포트 기간/제품 미리 선택해 즉시 미리보기 가능 상태로 준비
- 바코드 관리에서 보여줄 EPC 샘플 1건 사전 확보
- 브라우저 확대율 100%, 네트워크 탭 닫기, 알림창/팝업 차단 해제

9) 한 줄 마무리 멘트
==================================================
저희 프론트엔드는 "탐지 결과를 보여주는 화면"을 넘어서, 데이터 업로드부터 이상 분석, 조치 이력 관리, 운영 리소스 관리까지 현장에서 바로 쓰일 수 있는 실행형 워크플로우를 구현했습니다.
